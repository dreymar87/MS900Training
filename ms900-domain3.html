Microsoft Entra ID — Cloud Identity Platform

Microsoft Entra ID (formerly Azure Active Directory / Azure AD) is Microsoft's cloud-based identity and access management service. Every M365 tenant uses Entra ID as its identity backbone. It authenticates users and authorizes access to M365 apps, SaaS apps, and on-premises resources.

AuthenticationVerifying WHO you are (login)
AuthorizationVerifying WHAT you can access (permissions)
SSOSign in once, access all apps (Single Sign-On)
TenantYour org's dedicated Entra ID instance
📌
RBAC — Role-Based Access ControlEntra ID uses RBAC to limit what admins can do — assign only the least-privileged role needed. Key roles to know: Global Admin (full M365 control — use sparingly) · User Admin (create/edit users and assign licenses) · Billing Admin (manage subscriptions and payments) · Security Admin (configure Defender and Secure Score). Exam tip: a scenario asking "who should manage user accounts without touching billing?" → User Admin, not Global Admin.
Cloud, On-Premises & Hybrid Identity

Organizations can manage identities in three ways depending on their infrastructure.

Identity TypeWhere ManagedHow It Works
Cloud-onlyEntra ID (cloud)Users exist only in the cloud — no on-prem AD
On-premisesActive Directory (on-prem)Traditional AD DS; not cloud-connected
HybridBoth (synced)On-prem AD synced to Entra ID via Entra Connect; one identity for both worlds
📌
Exam tipMost enterprises use Hybrid identity — employees have one AD account that works on-prem AND in the cloud. Entra ID Connect is the sync tool that bridges them.
MFA, SSPR & Conditional Access

These three features work together to secure identities without creating excessive friction for users.

FeatureWhat It DoesWhy It Matters
MFA
(Multi-Factor Auth)		Requires 2+ verification factors: password + phone app / SMS / biometricsEven if password is stolen, attacker can't get in without the second factor
SSPR
(Self-Service Password Reset)		Users reset their own passwords using verified methods (email, phone, security questions)Reduces helpdesk tickets; users aren't locked out
Conditional AccessPolicy engine: "IF user is signing in from an unknown location, THEN require MFA"Contextual, risk-based access — tighter security with less friction for trusted scenarios
💡
The signals Conditional Access usesUser/group identity · Device compliance state · Location (IP/country) · Client app type · Sign-in risk level (from Entra ID Protection)
⚠️
Scenario trap — pick the right tool "User forgot their password and is locked out" → SSPR (self-service reset, no helpdesk needed). "Attacker has the user's password but can't get in" → MFA (second factor blocks them). "Block access from high-risk countries but allow trusted office IPs" → Conditional Access (context-based policy). The exam describes a scenario — know which tool solves which problem.

Microsoft Defender XDR — Unified Threat Protection

Defender XDR (Extended Detection and Response) is Microsoft's integrated security suite that correlates threat signals across all attack surfaces — endpoints, email, identity, and cloud apps — into a single view in the Microsoft Defender Portal.

Defender ProductProtectsKey Capability
Defender for EndpointDevices (Windows, macOS, Linux, mobile)Threat detection, EDR, vulnerability management
Defender for Office 365Email, Teams, SharePoint, OneDriveAnti-phishing, safe links, safe attachments
Defender for IdentityOn-premises AD identitiesDetects lateral movement, compromised credentials
Defender for Cloud AppsSaaS apps (Shadow IT)Discovers unsanctioned apps, enforces policies
💡
The XDR advantageInstead of each product working in isolation, Defender XDR correlates signals across all four pillars. An attack that starts with a phishing email → credential compromise → lateral movement is detected as one unified incident.
⚠️
Don't confuse these Defender products for Endpoint = Windows/Mac/Linux devices. for Office 365 = email & collaboration (phishing, malware in attachments). for Identity = on-premises Active Directory accounts (NOT cloud). for Cloud Apps = SaaS apps and shadow IT. Memory trick: the name tells you what it protects.
🖥️
The Microsoft Defender PortalAll four Defender products are managed from a single unified admin interface: security.microsoft.com — the Microsoft Defender Portal. Admins investigate incidents, review alerts, manage secure score, and run threat hunting from here. Separate from the Purview compliance portal (compliance.microsoft.com) and the M365 Admin Center (admin.microsoft.com).
Common Threats & How M365 Addresses Them

The exam tests your ability to map real-world attack types to the M365 tools that defend against them. Think of this as "if the threat is X, the answer is Y."

Threat TypeAttack VectorM365 Defence
Phishing / malicious emailFake emails trick users into clicking malicious links or attachmentsDefender for Office 365 — Safe Links, Safe Attachments, anti-phishing policies
Credential theft / password attacksBrute force, credential stuffing, or stolen passwordsMFA + Conditional Access + SSPR — second factor blocks access even with stolen passwords
Malware / ransomware on endpointsMalicious software installed on Windows/macOS devicesDefender for Endpoint — EDR, behavioural analysis, automated remediation
Compromised on-premises identityAttacker steals AD credentials and moves laterallyDefender for Identity — monitors on-prem AD for anomalous behaviour and lateral movement
Unsanctioned cloud apps (Shadow IT)Employees use unapproved SaaS apps that IT can't see or controlDefender for Cloud Apps — discovers shadow IT, enforces session controls and policies
Insider threats / data exfiltrationEmployees intentionally or accidentally leak sensitive dataPurview Insider Risk Management + DLP — detects and prevents data leaks
📌
Exam tipScenario questions often describe a threat and ask "which M365 tool should be used?" Learn the threat → tool mapping above cold. The most common traps: Defender for Identity = on-prem AD threats (not cloud); Defender for Cloud Apps = shadow IT/SaaS visibility (not endpoint protection).
Microsoft Secure Score

Secure Score is a numeric measurement (0–100+) of your organization's security posture in Microsoft 365. A higher score means more recommended security controls are in place. It provides actionable recommendations with estimated score improvements.

ScorePoints earned for security actions completed
RecommendationsSpecific actions to improve your score
ComparisonBenchmark vs. similar-sized orgs
TrendHistorical score tracking over time

Zero Trust Model

Zero Trust is a security philosophy that replaces the old "castle and moat" model. Rather than trusting everything inside your network, Zero Trust assumes breach and verifies every request explicitly.

🔍
Verify Explicitly
Always authenticate using all available data points (identity, location, device, service, workload)
🔒
Least Privilege
Limit user access with just-in-time (JIT) and just-enough-access (JEA)
🛡️
Assume Breach
Minimize blast radius. Segment access. Encrypt. Use analytics to detect threats.
📌
Zero Trust → M365 feature mapping The exam tests whether you can match Zero Trust principles to the specific M365 tools that implement them:

🔍 Verify Explicitly → MFA (verifies identity with multiple factors) + Conditional Access (evaluates every sign-in using identity, device, location, and risk signals)
🔒 Least Privilege Access → RBAC (admin roles scoped to minimum permissions) + Entra ID Privileged Identity Management (just-in-time access)
🛡️ Assume Breach → Defender XDR (detects breaches across all surfaces) + Purview eDiscovery & Auditing (investigate when breach occurs) + Sensitivity Labels (data protected even if stolen)

Memory hook: MFA = Verify. RBAC = Least Privilege. Defender = Assume Breach.
Microsoft Purview — Compliance Suite

Microsoft Purview is the umbrella brand for Microsoft's information protection and compliance tools. It helps organizations protect sensitive data, manage risk, and meet regulatory requirements.

Purview FeatureWhat It Does
Sensitivity LabelsClassify and protect documents/emails (e.g. "Confidential" label enforces encryption automatically)
Data Loss Prevention (DLP)Detects and prevents sharing of sensitive data (e.g. blocks emailing credit card numbers externally)
Insider Risk ManagementDetects risky user behaviors like data exfiltration or policy violations from within the org
eDiscoveryLegal hold and search across emails, Teams chats, SharePoint for litigation or investigation
AuditingUnified audit log of user and admin activities across M365 services
Data ResidencyEnsures customer data is stored in specific geographic regions for regulatory compliance
Compliance ManagerDashboard that scores your compliance posture against regulations (GDPR, ISO 27001, NIST) and provides improvement actions
📌
Compliance Manager vs Secure ScoreThese are two separate scores — easy to confuse. Secure Score (in the Defender Portal) measures your security posture. Compliance Manager (in the Purview portal) measures your regulatory compliance posture against specific frameworks like GDPR or ISO 27001. Both give you a score and a list of improvement actions, but they answer different questions.
Microsoft Priva — Privacy Management

Microsoft Priva helps organizations safeguard personal data and build a privacy-resilient workplace. It discovers where personal data lives across M365, identifies privacy risks, and enables subject rights request management.

Privacy Risk MgmtIdentifies risks like data overexposure or data transfers
Subject Rights RequestsAutomates GDPR/CCPA data access requests
Service Trust PortalMicrosoft's transparency hub — audit reports, compliance docs
⚠️
Purview vs Priva — easy to mix upPurview = protects organizational data (sensitivity labels, DLP, compliance, eDiscovery — think "protecting company secrets"). Priva = protects individuals' personal data (GDPR/CCPA rights requests, privacy risk — think "protecting people's private info"). Both live in the Purview portal, but they solve different problems.
